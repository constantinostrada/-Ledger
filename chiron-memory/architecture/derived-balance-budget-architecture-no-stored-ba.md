Derived-balance & budget architecture: no stored balances or spent totals

## Why
Eliminates drift by making the transaction ledger the single source of truth for all monetary aggregates

## Learned
Account.balance_cents column dropped; balance computed as credits−debits via Prisma groupBy aggregate over transactions in PrismaAccountRepository · Domain Account.balance is a hydrated read-only snapshot from the repository; canDebit checks run against this derived value · PrismaAccountRepository.deriveBalances computes both original-currency and base-currency balance in a single groupBy query via account→user relation, avoiding N+1 queries · initialBalanceCents materialized as an 'Opening balance' CREDIT transaction on account creation, not a stored field · budgets table has no spent_cents column; spent computed on-the-fly from period transactions via PrismaTransactionRepository.sumExpensesByCategory · budgets table stores limit_cents (Int) and period (Char(7) 'YYYY-MM') with unique constraint on (userId, categoryId, period) for exactly one limit per category per month · Budget entity enforces positive limitCents invariant in constructor/create() · Transaction entity enforces positive-amount invariant in create()/reconstitute() and carries optional categoryId with SetNull FK on category delete · All monetary fields stored as Int cents columns (balance_cents, amount_cents, base_amount_cents); DTOs use explicit *Cents names with zod .int() validation; floats rejected at HTTP edge and domain layer · Transactions store both original amount/currency and base_amount_cents/base_currency snapshot converted at posting time so aggregates never re-convert or depend on historical rates · Setting a budget is idempotent upsert (PUT /api/budgets, 200) on category+period; re-setting updates existing row · PUT chosen over POST to signal idempotent set-semantics for category+period upsert · SetBudgetUseCase checks category existence via ICategoryRepository before upserting, returning 404 for nonexistent categories · GetBudgetsUseCase computes spent for all budgets in a period with single batched sumExpensesByCategory call (all categoryIds at once), avoiding N+1 queries · ITransactionRepository extended with sumExpensesByCategory(userId, categoryIds, dateFrom, dateToExclusive) as single Prisma groupBy aggregate over EXPENSE transactions; uses null category key for uncategorized spend · sumExpensesByCategory enforces ownership through account→user relation in Prisma query itself, preventing cross-user category leakage · Budget/report aggregation sums base_amount_cents rather than amount_cents because spending categories span accounts in different currencies · ITransactionRepository exposes sumExpensesGroupedByCategory and sumByTypePerMonth read-side aggregate methods returning base-currency integer cents; reporting is pure read-side computation with no persistence, so aggregation logic lives behind repository interface · Category-grouped spend uses Prisma groupBy, but monthly income/expense series uses raw SQL with date_trunc('month', ...) because Prisma groupBy cannot group by derived/truncated date expressions · Raw SQL monthly series must use positional GROUP BY (GROUP BY 1, 2) instead of GROUP BY month, type because both transactions and accounts tables have a column named type, causing Postgres ambiguity error (caught only by manual curl end-to-end run, not unit tests with fake repositories) · Report endpoints (spend-by-category, income-vs-expense, net-worth) each resolve user's base currency once and perform single aggregate query per use case, avoiding N+1 queries · Category is a per-user owned entity with required userId FK (cascade delete), kind (INCOME/EXPENSE reusing TransactionType enum), and color (hex VARCHAR(7)); added in B1 with ICategoryRepository port and PrismaCategoryRepository implementation alongside pre-existing Account/Transaction ports · Deleting a category is blocked with 409 if referenced by any transaction, recurring rule, or budget; no auto-reassignment · Use cases accepting categoryId (CreateTransactionUseCase, CreateRecurringRuleUseCase, SetBudgetUseCase) enforce category ownership by masking foreign/nonexistent category as generic 'not found' error, same pattern as account ownership checks · Category create/rename duplicate-name checks mirror register-email uniqueness pattern (check-then-guard in use case, not just DB constraint); self-rename (same name, same category) explicitly allowed · Each account holds a single currency; transaction's currency must match its account's currency or TransactionService rejects it; recurring rule's currency must match its account's currency (CreateRecurringRuleUseCase) to keep single-currency-per-account invariant for materialized recurring transactions · Multi-currency migration backfilled base_amount_cents/base_currency on all pre-existing transaction rows 1:1 since all prior data was USD-only · RecurringRule ownership checked via its account relation (account_id → account.user), same not-found-error pattern as transactions, rather than storing direct user/tenant id on rule · When resource (account/transaction/category) doesn't belong to requesting user, use cases return 'not found' rather than 403 to avoid leaking existence of other users' resources; cross-tenant/cross-user access checks look identical to genuine not-found case · Posting an expense that would make balance negative rejected with 'Insufficient funds for expense transaction'; balance updates immediately on transaction posting
